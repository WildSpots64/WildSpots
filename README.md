# WildSpots — démo

Landing page + carte interactive de 25 spots de surf, kite, wind et wingfoil dans les Landes et le Pays Basque.

## Publier sur GitHub Pages (gratuit, ~5 min)

1. Va sur **github.com** et crée un nouveau repository (par ex. `wildspots-demo`), public.
2. Sur la page du repo, clique **Add file → Upload files**, puis dépose les deux fichiers de ce dossier :
   - `index.html`
   - `og-image.png`
3. Valide le commit (**Commit changes**).
4. Va dans **Settings → Pages** (menu de gauche).
5. Sous **Build and deployment**, choisis **Source : Deploy from a branch**, puis **Branch : main** et dossier **/ (root)**. Clique **Save**.
6. Attends 1-2 minutes : GitHub affiche l'URL publique en haut de la page, du type :
   `https://TON-PSEUDO.github.io/wildspots-demo/`

C'est cette URL qu'il faut partager sur WhatsApp, Facebook, etc. — l'aperçu (image + titre) s'affichera automatiquement grâce aux balises Open Graph déjà présentes dans `index.html`.

## Mettre à jour le site ensuite

Pour éditer le contenu (nouveaux spots, textes, design) :
- Soit tu modifies `index.html` directement sur GitHub (bouton crayon ✏️ sur la page du fichier), commit → le site se met à jour automatiquement en ~1 min.
- Soit tu me redonnes le fichier ici, je fais les modifications, et tu re-uploads la nouvelle version sur GitHub (même procédure qu'à l'étape 2, ça écrase l'ancien fichier).

## Activer la sauvegarde réelle des avis et validations (Supabase, gratuit)

Sans cette étape, les avis et les ⭐ restent dans le navigateur de chaque visiteur et disparaissent au rechargement. Avec Supabase, ils sont enregistrés pour de vrai et visibles par tout le monde.

1. Va sur **supabase.com** → **Start your project** → crée un compte (gratuit) → **New project**. Choisis un nom, un mot de passe de base de données (à garder de côté), une région proche (Europe).
2. Une fois le projet créé (~2 min), va dans **SQL Editor** (menu de gauche) → **New query**, colle ce script, puis **Run** :

```sql
create table comments (
  id bigint generated by default as identity primary key,
  spot_index int not null,
  spot_name text,
  author text,
  comment text not null,
  created_at timestamptz default now()
);

create table validations (
  id bigint generated by default as identity primary key,
  spot_index int not null,
  spot_name text,
  section_key text not null,
  created_at timestamptz default now()
);

alter table comments enable row level security;
alter table validations enable row level security;

create policy "Public read comments" on comments for select using (true);
create policy "Public insert comments" on comments for insert with check (true);

create policy "Public read validations" on validations for select using (true);
create policy "Public insert validations" on validations for insert with check (true);
```

3. Va dans **Settings → API** (menu de gauche). Copie deux valeurs :
   - **Project URL** (ex. `https://xxxxx.supabase.co`)
   - **anon public** key (une longue chaîne de caractères — c'est une clé publique, normal qu'elle soit visible dans le code du site)
4. Ouvre `index.html`, cherche ces deux lignes tout en haut du script (`Ctrl+F` "COLLE_TON") :
   ```js
   const SUPABASE_URL = "COLLE_TON_URL_SUPABASE_ICI";
   const SUPABASE_ANON_KEY = "COLLE_TA_CLE_ANON_ICI";
   ```
   Remplace les deux valeurs par celles copiées à l'étape 3.
5. Réupload `index.html` sur GitHub (même procédure qu'à l'étape 2 de publication) — les avis et validations sont maintenant partagés entre tous les visiteurs.

**Limite à connaître pour un vrai pilote** : n'importe quel visiteur peut poster un commentaire ou valider sans compte (pas d'authentification). C'est volontairement simple pour un pilote — ça permet de tester vite sans friction d'inscription — mais ça veut dire pas de modération automatique ni de protection contre le spam. Si le pilote se passe bien et que tu passes à plus de volume, il faudra ajouter une authentification légère (email, ou connexion via Google) et des règles de modération.
